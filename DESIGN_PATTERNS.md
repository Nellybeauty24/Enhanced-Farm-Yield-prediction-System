# Design Patterns Documentation: Soil Nutrition Backend

This document details the software design patterns implemented in the Soil Nutrition Backend to ensure scalability, maintainability, and clean code.

---

## 1. Creational Patterns

### App Factory Pattern
The application uses the **App Factory** pattern in `app/__init__.py:create_app()`.
- **Purpose**: Allows for multiple application instances with different configurations (e.g., testing vs. production).
- **Implementation**: `create_app` initializes Flask, registers extensions (DB, CORS), and attaches Blueprints.

### Singleton Pattern
The `CropPredictionService` class in `app/services/crop_service.py` implements the **Singleton** pattern.
- **Purpose**: ML models (Pickle, CatBoost) are large and expensive to load. This ensures they are loaded only once and reside in memory for the lifetime of the application.
- **Implementation**: Using `__new__` to return the same instance across all service calls.

---

## 2. Structural Patterns

### Blueprint Pattern
Routes are organized into **Blueprints** (e.g., `prediction`, `analytics`, `auth`).
- **Purpose**: Decouples independent features of the application, making the codebase easier to navigate and scale.
- **Implementation**: Each module defines its own routes and is registered in the main app factory.

### Decorator Pattern
Security is enforced using the **Decorator** pattern with `@require_auth`.
- **Purpose**: To dry (Don't Repeat Yourself) the authentication logic across multiple routes.
- **Implementation**: Wraps endpoint functions to check for valid JWTs before execution.

---

## 3. Behavioral Patterns

### Service Layer Pattern
Business logic is extracted from the API controllers into a dedicated **Service Layer** (`app/services/`).
- **Purpose**: Separates concerns. API controllers focus on HTTP handling (status codes, JSON), while services focus on the "what" and "how" of prediction and recommendation.
- **Implementation**: `CropPredictionService` handles the ML model interactions, while `ReservationService` handles the farming rules.

### Data Transfer Object (DTO) / Validation Pattern
The application uses **Marshmallow Schemas** as DTOs between the client and the server.
- **Purpose**: Ensures that only clean, validated, and safely-typed data enters the business logic layer.
- **Implementation**: `PredictionInputSchema` validates and cleans the raw JSON before it is passed to the ML prediction services.

---

## 4. Key Architectural Decisions

- **ORM (Object-Relational Mapping)**: Using SQLAlchemy to treat database records as Python objects, simplifying complex queries in `analytics.py`.
- **Stateless Authentication**: Using JWTs instead of server-side sessions, making the API more scalable and easier to integrate with mobile applications.
